## Executive Summary

The recent ruthless benchmark results showed 0% doctrine accuracy, 0% citations, 0% faithfulness, and a p95 latency of 161 seconds. Our forensic analysis has identified three primary bugs that completely explain these failures:

1. **RAG Pipeline Crash due to NameError (100% of Doctrine Failures):** The RAG pipeline was raising an unhandled `NameError` in `retrieve_documents` because the `query_tier` variable was referenced without being defined. The outer endpoint handler caught this exception and returned the default apology: *"I apologize, but I don't have that specific teaching."*
2. **Double Prefixing on Query Embeddings (0% Retrieval Accuracy):** The query embedding pipeline prepended the query text with the E5 instruction prompt twice, producing garbage cosine similarity comparison scores against a corpus indexed with only a single prefix.
3. **Cumulative Node Timeout Retries (150+ Second p95 Latency):** Under-budgeted timeouts (15s) forced the slow reasoning model (`sarvam-30b`, used for both generation and fast classification) to time out and trigger maximum retries (3 attempts) on early nodes, creating a massive latency cliff before the NameError occurred.

## 1. Bug Audits & Diagnostics

### Bug A: NameError on `query_tier` in `retrieve_documents`
* **File:** [nodes.py](file:///Users/harshodaikolluru/Public/askmukthiguru-8119b0e8/backend/rag/nodes.py#L1124)
* **Code snippet:**
  ```python
  _lightrag if query_tier != "fast" else None,
  ```
* **Root Cause:** The `query_tier` variable is referenced to determine if LightRAG should be bypassed on fast paths, but it is never extracted from the GraphState input `state`.
* **Impact:** Every query routed through the `standard` or `deep` graphs crashed instantly at the retrieval node. Uvicorn logs captured the following exception:
  `NameError: name 'query_tier' is not defined`
  FastAPI caught this error and returned the default fallback: *"I apologize, but I don't have that specific teaching. 🙏 Please try asking another question."* with an `intent` of `"ERROR"`.

### Bug B: Double Query Prefixing
* **File:** [embedding_service.py](file:///Users/harshodaikolluru/Public/askmukthiguru-8119b0e8/backend/services/embedding_service.py#L297-L316)
* **Root Cause:**
  1. `encode_single_full(text)` prepends `self.instruction` to the query:
     `prefixed_text = f"{self.instruction}{text}"`
  2. It then calls `self.encode_batch([prefixed_text])`.
  3. `encode_batch` loops through the input text array and prepends `self.instruction` *again*:
     `prefixed_text = f"{self.instruction}{text}"`
* **Impact:** The query vector is generated on a double-prefixed string:  
  `"Given a spiritual teaching, retrieve relevant passages: Given a spiritual teaching, retrieve relevant passages: <query>"`  
  This mismatched the corpus chunks which are indexed with a single prefix. Because E5 models are highly sensitive to instruction structures, all cosine similarity search scores degraded to noise.

### Bug C: Cumulative Timeout and Retry Loop
* **File:** [config.py](file:///Users/harshodaikolluru/Public/askmukthiguru-8119b0e8/backend/app/config.py) & [sarvam_service.py](file:///Users/harshodaikolluru/Public/askmukthiguru-8119b0e8/backend/services/sarvam_service.py)
* **Root Cause:** The reasoning model `sarvam-30b` was selected for both generation and fast classification (`sarvam_cloud_classify_model = "sarvam-30b"`). A reasoning model spends significant time generating internal thoughts before producing content. 
* **Impact:** Fast nodes like `intent_router` and `decompose_query` have a tight 15s timeout limit. Under load, these calls timed out with `ReadTimeout` / `CancelledError`. Because `llm_max_retries = 2`, the backend performed 3 attempts per node, compounding to `35s * 3 = 105s` on a single node before either failing or continuing, causing the overall latency to hit the 150s floor.

## 2. Qdrant Collection Audit (`spiritual_wisdom`)

* **Endpoint `/collections/spiritual_wisdom` Info:**
  - `status`: `"grey"`
  - `optimizer_status`: `"ok"`
  - `vectors_config`: `dense` (1024d Cosine) and `sparse` (named `sparse`)
  - `points_count`: 94,634
* **Explanation for Grey Status:** The collection was indexed using `intfloat/multilingual-e5-large-instruct`. The local service container loads this model via `FlagEmbedding.BGEM3FlagModel` which is hardcoded for BGE-M3. Since E5 has no sparse head, `lexical_weights` returns `defaultdict(int, {})` (empty dict). 
* **Payload Verification:** As a result, the collection contains exactly **zero** sparse vector components. Qdrant marks the collection as `"grey"` (inactive sparse optimizer index status) because a sparse vector field is declared in the schema but no sparse indexes/vectors are actively populated. The dense vector HNSW index remains fully active.

## 3. Recommended Remediation Order

1. **Fix the NameError:** Extract `query_tier = state.get("query_tier")` in `retrieve_documents`.
2. **Fix Double Prefixing:** Eliminate the double `self.instruction` prefix in `EmbeddingService.encode_single_full` by passing raw text to `encode_batch` or bypassing extra prefixing.
3. **Configure LLM timeouts & retries:** Cap CRAG loops and add recursion limits to LangGraph. Adjust fast-call timeouts to allow `sarvam-30b` to generate reasoning content safely when used, or configure a faster classification model.
4. **Fix Test Suite Mocking:** Update `tests/test_chat_endpoint.py` mock container to match the tiered routing graph attributes (`standard_graph`, `fast_graph`, `deep_graph`) to resolve pytest failures.
