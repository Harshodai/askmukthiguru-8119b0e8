**Skill context:** mobile release for a Capacitor app. Your
`docs/MOBILE_RELEASE_RUNBOOK.md` already covers the *manual* steps well —
this pack adds the **three things that make a first deploy feel safe**:
(1) one-command reproducible builds via Fastlane, (2) ready-to-paste store

Your setup (verified from the repo): Capacitor 8, `appId
com.askmukthiguru.app`, plugins: app, device, preferences, push-notifications.
`android/` + `ios/` are gitignored (generated by `npx cap add`).

## 0. The 20-minute confidence quickstart

Do a **dry run on the internal track first** — you can iterate freely there
with zero user impact. This is the single best cure for deploy anxiety.

```bash
# 1. build the web bundle (with your prod env)
npm run build                      # vite build + SEO prerender

# 2. generate native projects (one-time)
npx cap add android
npx cap add ios
npx cap sync                        # copies dist/ into both

# 3. Android: internal-track AAB
cd android && ./gradlew bundleRelease
# → android/app/build/outputs/bundle/release/app-release.aab

# 4. iOS: open Xcode, archive, upload to TestFlight
npx cap open ios
```

If steps 1–4 succeed on your machine, you *can* ship — everything after is
store paperwork, which this pack pre-fills for you.

## 1. Fastlane (one-command, reproducible)

Removes the "did I click the right thing in Android Studio/Xcode?" fear.

**Android** — `android/fastlane/Fastfile` (see `fastlane-android-Fastfile` in
this folder):
```bash
cd android
fastlane internal     # build + upload AAB to Play internal track
fastlane release      # promote to production
```

**iOS** — `ios/fastlane/Fastfile` (see `fastlane-ios-Fastfile`):
```bash
cd ios
fastlane beta         # build + upload to TestFlight
fastlane release      # submit to App Store review
```

Setup once: `gem install fastlane`, `fastlane init` in each platform dir, put
your Play service-account JSON at `android/fastlane/play-console.json` and
your App Store Connect API key at `ios/fastlane/AuthKey.p8`.

Reflections are **linked to the user** through the account/vault
relationship regardless of encryption, because the vault is associated with

current inference path uses local Ollama models and does not call external

## 5. The honest risks (so nothing surprises you)

1. **Apple review may ask about account deletion** — have the in-app delete
   (crypto-shred) visible in settings before you submit. This is the #1
   rejection reason for apps like yours right now.
2. **Capacitor + service worker caching** — after an update, users can see a
   stale bundle. Your `main.tsx` already handles SW updates; verify the
   "new version available" prompt appears on device.
3. **Push on iOS needs a real APNs key** — TestFlight push won't work with a
   simulator; test on a physical device.
4. **First submission takes 24–48h review** — plan launch comms accordingly;
   internal/closed tracks are instant.
